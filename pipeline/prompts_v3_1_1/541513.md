## What you author

Write one JSON object conforming exactly to `pipeline/build/schemas/research_draft_v3_1_1.schema.json`. You author reusable atomic `evidence_facts` and selected research/judgment inputs. You do **not** author final provenance, deterministic dataset values, role weights, role contributions, factor scores, `S`, uplift or a verdict. Plain Python owns those fields.

Every selected input has a `method`:

- `OBSERVED` — `value` repeats a value stated by a cited fact exactly. Include the identical `observed_value`. No interpretation, proxy, normalization or judgmental bridge is permitted.
- `CALCULATED` — `value` is fully reproducible arithmetic over named numeric operands, every operand is backed by a cited fact, and the expression uses only numeric constants, parentheses and `+ - * /`. Include every operand's `name`, `value` and `fact_id`. Python recomputes the result. Any judgmental mapping makes this method invalid.
- `ESTIMATE` — required whenever any bridge from facts to the selected value is judgmental, including proxy choice, population mapping, normalization assumptions or an opinion. Its structured `estimate_basis` must enumerate every starting fact ID, every scope mismatch, each ordered proxy/normalization/population step, and the explicit judgmental step reaching the selected value. A cited background fact does not turn an estimate into `OBSERVED` or `CALCULATED`.

Python maps `OBSERVED → DIRECT`, `CALCULATED → DERIVED`, and `ESTIMATE → ESTIMATE`. Never author `provenance_type` in the draft. `evidence_quality` is a separate reliability axis: `HIGH | MED | LOW | NONE`. Empty `fact_ids` requires `NONE`; non-empty `fact_ids` requires non-`NONE` quality.

## Citation discipline — C1 to C4

1. **C1 — Fetch before cite.** Open and inspect the exact page before creating a fact. Search snippets, titles and memory are not evidence.
2. **C2 — Quote what you saw.** Each fact contains one short exact contiguous quote or table figure. Compact (`...`), spaced (`. . .`) and Unicode (`…`) ellipses are forbidden. Never splice separated passages.
3. **C3 — Attribute only what the URL establishes.** Never use a URL as support for a claim it does not establish. If no fetched source supports a background fact, omit the fact and label the selected value honestly as an estimate.
4. **C4 — Reopen and verify.** Before submission, enumerate and reopen every URL and verify its quote, attribution and all structured scope fields against the live page.

`evidence_facts[]` is the only place URLs may appear. Use stable unique IDs `F1`, `F2`, ... and reuse a fact by ID instead of duplicating its URL/quote. Every scope must state `population`, `geography`, `unit`, `period` and `industry_mismatch`. If a source omits any scope element, write exactly `not reported` for that field; never infer it silently.

## Frozen research rules

1. PE/M&A activity informs only B and M, never V or A.
2. Multiples are EV/EBITDA after market-rate owner compensation. State unit, target size, source population and every normalization. A normalization or market mapping that requires judgment is `ESTIMATE`.
3. A zero consolidator count is an absence-of-evidence estimate, not an observation. State the searches performed; do not cite a topical page as proving zero.
4. `current_adoption_pct` means the share of US firms in this NAICS materially using AI in automatable functions. Tool use, worker use and broad-sector use are different populations. Any judgmental mapping to the target population is `ESTIMATE`.
5. `t50_years` means years from today until 50% material firm adoption. If current adoption is already at least 50%, set it to `0`; otherwise it must be positive. Forecasting or analog-based bridging is normally `ESTIMATE`, even when supported by observed adoption facts.
6. Payer/regulatory repricing belongs in C when it determines who retains cost savings. Terminal class separately answers whether paid end demand survives mature AI; do not count the same clawback twice.
7. A statutory human-in-the-loop boundary affects terminal class only to the extent it preserves paid demand. It does not make automatable tasks non-automatable.
8. If a W-2 dataset labor share is distorted by 1099-heavy labor, use the injected value unchanged, flag `dataset_mismatch`, lower confidence as appropriate and show sensitivity. Do not silently adjust it.
9. Catch-all codes: identify at least the top three revenue subsegments, score the dominant subsegment, set `heterogeneous=true`, and complete the structured market-boundary disclosure.
10. `succession_shortage_documented` changes B. It is a full boolean selection under the same method/fact contract. `true` requires `OBSERVED`, at least one fact that directly documents a shortage, and exact live verification. A search-based `false` is normally `ESTIMATE` + `NONE`, with the searches and absence judgment fully disclosed in `estimate_basis`.

## Industry-specific guidance — hints, not facts

**Likely role structure:**
- **Operations engineers and administrators (the delivery core)** — ~30-35% of wage bill (OEWS basis: 15-1252 Software Developers 23.9% at 4-digit 5415 — for this code much of the technical wage bill is really systems/network administrators (15-1244) and operations staff in the unlisted tail; est. ~30-35% combined); automatable: partial. Monitoring, patching, incident triage, capacity reporting and runbook execution are the exact target of AIOps/self-healing tooling — high within-role fraction; physical data-center hands (rack/stack, media handling) and client-site presence persist.
- **Support and helpdesk staff** — ~6-10% of wage bill (OEWS basis: 15-1232 Computer User Support Specialists 3.3% at 4-digit; likely higher in facilities-management contracts, est. ~6-10%); automatable: yes. Tier-1/2 support under outsourcing contracts — directly AI-automatable; contracts often price it per ticket or per seat, so automation economics hinge on contract form.
- **Systems analysts** — ~7% of wage bill (OEWS basis: 15-1211 Computer Systems Analysts 4.6% + 13-1111 Management Analysts 2.7%); automatable: partial. Service-level reporting, transition planning docs and audits automate; client governance interaction persists.
- **Service delivery and account management** — ~11% of wage bill (OEWS basis: 11-3021 Computer & Information Systems Managers 8.3% + 13-1082 Project Management Specialists 3.0%); automatable: partial. SLA reporting and coordination automate; the contract-governance relationship with client IT leadership persists and is the retention asset.
- **Company management and sales** — ~9-10% of wage bill (OEWS basis: 11-1021 General & Operations Managers 5.2% + 41-3091 Sales Reps 4.3%); automatable: partial. Long-cycle outsourcing-contract sales and renewals persist; internal ops automate.
- **Admin and unallocated tail** — ~25-30% of wage bill (OEWS basis: remainder beyond the listed top-10 (top-10 sums to ~61% at 4-digit), est. ~25-30% — includes shift operators and physical facilities staff); automatable: partial. Back office automates; shift operations staff are the AIOps target; physical facilities work is not automatable.

**Capture questions:**
- Facilities-management/outsourcing contracts price as multi-year fixed fees, per-device/per-seat rates, or cost-plus with committed year-over-year productivity glidepaths — under these structures, automation savings are typically PRE-COMMITTED to the client at renewal (the Kyndryl/DXC pattern of shrinking contract value). In the $1-10M EBITDA tier, what contract forms dominate, and is there any evidence operators retain automation gains past one renewal cycle?
- Who owns the client relationship — the firm holds multi-year contracts with real switching costs (transition risk keeps clients captive), but renewals are procurement-run and benchmarked; does incumbency pricing power survive a benchmarking clause?
- Cloud migration is the structural bypass: clients don't need generic AI to leave — they need AWS/Azure, which has been shrinking the on-prem facilities-management base for a decade; what is the evidenced trajectory of the addressable base, and does AI accelerate workload migration off managed on-prem estates?
- For the surviving base (regulated, legacy, sovereignty-bound workloads), does compliance/audit accountability create durable fees that AI operations tooling cannot displace — and how big is that residue?

**Terminal-value question:**
This code sells the ongoing human operation of client IT estates — precisely the function that both cloud (structurally, for a decade) and AIOps (now) replace. The mechanism to check is contractual: outsourcing renewals with benchmarking clauses and productivity commitments hand automation gains to clients on a fixed cycle, while the addressable estate itself migrates to hyperscalers whose operations are already automated. Is there a defensible terminal niche (regulated/legacy/sovereign workloads with audit-accountability fees), or is this a melting-ice-cube base where AI is not the operator's tool but the client's exit ramp? Score the niche, not the 2022 snapshot.

**Known traps:**
- Labor share 0.3989 is the group's lowest for a reason: receipts carry pass-through hosting, hardware, software licenses and subcontracted capacity — normalize before inferring margin structure, and expect wide variance between labor-based contracts and pass-through-heavy ones.
- Sibling boundaries: 541513 = ongoing management/operation of a CLIENT'S computer systems and facilities (outsourced IT ops, data-center operation for the client); 541512 = designing/integrating systems and the SMB MSP model; 541511 = custom programming; 541519 = residual incl. disaster recovery. Also outside the family: 518210 (computing infrastructure providers/hosting — the provider owns the infrastructure) contaminates market data constantly; if the firm owns the data center, it is not 541513.
- Size-structure warning: this code is big-firm skewed (151 firms at 500+, avg $117M revenue) — the fragmented-small-firm roll-up pattern fits poorly; n_band 583 exists but verify how many are independent firms vs. subsidiaries/captives before scoring B.
- Secular headwind predates AI: cloud migration has been shrinking traditional facilities management since the 2010s — single-vintage (2022) receipts overstate the durable base; check the trend, and do not attribute cloud-driven decline to AI or vice versa.
- Contract-form deflation is the central capture fact: benchmarking clauses and productivity glidepaths in outsourcing contracts are explicit, documented mechanisms that pass efficiency gains to clients — model pi_dist off contract structure, not off generic services intuition.

## Frozen formulas — Python calculates them

- `V_raw = labor_share × ai_replaceable_share`
- `V = 10 × min(1, V_raw / 0.25)`
- `C = 10 × pi_dist × pi_moat`
- `A = 10` when `t50_years=0`, otherwise `min(10, 10 / t50_years)`
- `TD = clamp(log10(n_band) / 4, 0, 1)`
- `OW = clamp((owners_60plus_pct − 0.10) / 0.35, 0.1, 0.9)`; add 0.1 for documented succession shortage, cap at 1
- `CFD = min(0.9, log10(1 + active_consolidators) / 2)`
- `B = 10 × sqrt(TD × OW) / (1 + 0.3 × CFD)`
- `M = clamp(4 × (exit_mult − buy_mult) / buy_mult, 0, 10)`
- `S = (V × C × A × B × M)^(1/5)`
