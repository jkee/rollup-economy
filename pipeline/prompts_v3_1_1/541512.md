## What you author

Write one JSON object conforming exactly to `pipeline/build/schemas/research_draft_v3_1_1.schema.json`. You author reusable atomic `evidence_facts` and selected research/judgment inputs. You do **not** author final provenance, deterministic dataset values, role weights, role contributions, factor scores, `S`, uplift or a verdict. Plain Python owns those fields.

Every selected input has a `method`:

- `OBSERVED` — `value` repeats a value stated by a cited fact exactly. Include the identical `observed_value`. No interpretation, proxy, normalization or judgmental bridge is permitted.
- `CALCULATED` — `value` is fully reproducible arithmetic over named numeric operands, every operand is backed by a cited fact, and the expression uses only numeric constants, parentheses and `+ - * /`. Include every operand's `name`, `value` and `fact_id`. Python recomputes the result. Any judgmental mapping makes this method invalid.
- `ESTIMATE` — required whenever any bridge from facts to the selected value is judgmental, including proxy choice, population mapping, normalization assumptions or an opinion. Its structured `estimate_basis` must enumerate every starting fact ID, every scope mismatch, each ordered proxy/normalization/population step, and the explicit judgmental step reaching the selected value. A cited background fact does not turn an estimate into `OBSERVED` or `CALCULATED`.

Python maps `OBSERVED → DIRECT`, `CALCULATED → DERIVED`, and `ESTIMATE → ESTIMATE`. Never author `provenance_type` in the draft. `evidence_quality` is a separate reliability axis: `HIGH | MED | LOW | NONE`. Empty `fact_ids` requires `NONE`; non-empty `fact_ids` requires non-`NONE` quality.

## Citation discipline — C1 to C4

1. **C1 — Fetch before cite.** Open and inspect the exact page before creating a fact. Search snippets, titles and memory are not evidence.
2. **C2 — Quote what you saw.** Each fact contains one short exact contiguous quote or table figure. Compact (`...`), spaced (`. . .`) and Unicode (`…`) ellipses are forbidden. Never splice separated passages.
3. **C3 — Attribute only what the URL establishes.** Never use a URL as support for a claim it does not establish. If no fetched source supports a background fact, omit the fact and label the selected value honestly as an estimate.
4. **C4 — Reopen and verify.** Before submission, enumerate and reopen every URL and verify its quote, attribution and all structured scope fields against the live page.

`evidence_facts[]` is the only place URLs may appear. Use stable unique IDs `F1`, `F2`, ... and reuse a fact by ID instead of duplicating its URL/quote. Every scope must state `population`, `geography`, `unit`, `period` and `industry_mismatch`. If a source omits any scope element, write exactly `not reported` for that field; never infer it silently.

## Frozen research rules

1. PE/M&A activity informs only B and M, never V or A.
2. Multiples are EV/EBITDA after market-rate owner compensation. State unit, target size, source population and every normalization. A normalization or market mapping that requires judgment is `ESTIMATE`.
3. A zero consolidator count is an absence-of-evidence estimate, not an observation. State the searches performed; do not cite a topical page as proving zero.
4. `current_adoption_pct` means the share of US firms in this NAICS materially using AI in automatable functions. Tool use, worker use and broad-sector use are different populations. Any judgmental mapping to the target population is `ESTIMATE`.
5. `t50_years` means years from today until 50% material firm adoption. If current adoption is already at least 50%, set it to `0`; otherwise it must be positive. Forecasting or analog-based bridging is normally `ESTIMATE`, even when supported by observed adoption facts.
6. Payer/regulatory repricing belongs in C when it determines who retains cost savings. Terminal class separately answers whether paid end demand survives mature AI; do not count the same clawback twice.
7. A statutory human-in-the-loop boundary affects terminal class only to the extent it preserves paid demand. It does not make automatable tasks non-automatable.
8. If a W-2 dataset labor share is distorted by 1099-heavy labor, use the injected value unchanged, flag `dataset_mismatch`, lower confidence as appropriate and show sensitivity. Do not silently adjust it.
9. Catch-all codes: identify at least the top three revenue subsegments, score the dominant subsegment, set `heterogeneous=true`, and complete the structured market-boundary disclosure.
10. `succession_shortage_documented` changes B. It is a full boolean selection under the same method/fact contract. `true` requires `OBSERVED`, at least one fact that directly documents a shortage, and exact live verification. A search-based `false` is normally `ESTIMATE` + `NONE`, with the searches and absence judgment fully disclosed in `estimate_basis`.

## Industry-specific guidance — hints, not facts

**Likely role structure:**
- **Engineers and technical delivery staff (developers, systems engineers)** — ~28-30% of wage bill (OEWS basis: 15-1252 Software Developers 23.9% of wage bill + 15-1299 Computer Occupations All Other 3.1% + 15-1253 Software QA 2.2% (role_mix is 4-digit 5415; for this code — systems design/integration/MSP — some of the 'developer' share is really systems/network engineers)); automatable: partial. Scripting, integration glue code, configuration, patching automation, runbook drafting and documentation are heavily AI-exposed; solution architecture judgment, client-specific environment knowledge and on-site/hardware work persist. For MSP-model firms, read this role as centralized services engineers — the tier where AI automation (self-healing endpoints, automated remediation) is already commercial.
- **Helpdesk and support desk** — ~8-12% (record likely understates) of wage bill (OEWS basis: 15-1232 Computer User Support Specialists 3.3% at 4-digit — materially understated for MSP-heavy 541512, where support desks are the largest headcount block; est. ~8-12% at 6-digit); automatable: yes. Tier-1 ticket resolution, password resets, onboarding/offboarding — the most directly AI-automatable cost line in an MSP, and the one AI service-desk products already target. Physical desk-side/field visits are the non-automatable residue.
- **Systems analysts and consultants** — ~7% of wage bill (OEWS basis: 15-1211 Computer Systems Analysts 4.6% + 13-1111 Management Analysts 2.7%); automatable: partial. Assessments, migration planning docs and compliance gap analyses automate substantially; stakeholder work and vendor selection judgment persist.
- **Technical and delivery management** — ~11% of wage bill (OEWS basis: 11-3021 Computer & Information Systems Managers 8.3% + 13-1082 Project Management Specialists 3.0%); automatable: partial. vCIO/technology-alignment reviews, project coordination and reporting automate in part; the trusted-advisor QBR relationship is the retention mechanism and persists.
- **Company management** — ~5% of wage bill (OEWS basis: 11-1021 General & Operations Managers 5.2%); automatable: partial. Owner-operators hold the client relationships in sub-$10M firms; internal ops automate.
- **Sales and account management** — ~4-5% of wage bill (OEWS basis: 41-3091 Sales Representatives of Services 4.3%); automatable: partial. Proposal/quote production and renewals paperwork automate; local relationship selling to SMB owners persists.
- **Admin, dispatch, finance and unallocated tail** — ~28-32% of wage bill (OEWS basis: long tail beyond the listed top-10 (top-10 sums to ~61% of wages), est. remainder ~28-32%); automatable: partial. Ticketing admin, dispatch, procurement/quoting, invoicing — heavily automatable; part of the tail is additional field/bench technicians (physical) — allocate between patterns.

**Capture questions:**
- The MSP model bills per-seat or per-endpoint monthly recurring fees under 1-3 year agreements — when AI service desks and self-healing endpoint automation cut delivery cost per seat, the operator keeps the spread as margin until competitive repricing; but the same billing unit creates a second channel: if AI shrinks client headcount, billed seats shrink with it. What is the evidence on both — are per-seat rates holding or compressing in competitive renewals, and are seat counts at SMB clients starting to decline?
- Who owns the client relationship — the firm (contract, documentation, QBR cadence, tooling lock-in) or the owner/lead engineer personally? MSP agreements, environment documentation and security tooling create real switching costs; assess how much survives an acquisition and whether platform roll-ups' retention data confirms it.
- The RMM/PSA/security vendors (Kaseya, ConnectWise, Microsoft) sit upstream of every MSP — as they embed autonomous remediation and AI service desks into the platforms themselves, do they capture the automation value via license pricing (squeezing MSP gross margin), or even sell AI-run IT management directly to SMBs, disintermediating the MSP layer?
- What realistically stops an SMB from bypassing its MSP with generic AI plus SaaS defaults — is the durable moat security/compliance accountability (cyber insurance requirements, CMMC/HIPAA attestations, incident response liability) rather than routine IT labor, and what share of MSP revenue is already security/compliance-anchored?
- Hardware resale and project work (migrations, installs) still make up a large minority of small-MSP revenue — those are competitively bid and pass AI savings through; what is the MRR vs. project/resale split in the target tier, since capture economics differ completely between them?

**Terminal-value question:**
SMBs will still need their IT to run, but the 5-10 year question is who operates the AI that runs it: if the MSP becomes the accountable operator of autonomous IT/security agents for clients too small to do it themselves, AI cuts the operator's costs and per-seat MRR holds as an insurance-like fee; if instead the RMM/security platform vendors ship autonomous management that SMBs consume directly (or AI collapses SMB software estates into a few self-managing SaaS suites), the MSP's product is replaced and the seat base it bills evaporates. The deciding mechanisms are concrete — cyber-insurance and compliance attestation requirements naming an accountable provider, and platform vendors' channel economics (do Kaseya/ConnectWise/Microsoft need the MSP channel or compete with it?). Which way does current evidence point, and does per-seat billing survive contact with AI-shrunk client headcounts?

**Known traps:**
- This code hosts the flagship MSP roll-up wave: ~169+ disclosed MSP deals in 2025 with PE in ~69%, platforms (Evergreen, New Charter, Ntiva, Thrive, Integris) buying at 4-7x small-firm EBITDA and trading at 10-13x — prime evidence for B and M, and per rule 2 absolutely inadmissible as evidence for V or A. Keep the wave out of automatability and adoption scoring.
- Sibling boundaries (the run WILL hit contaminated data): 541512 = designing and integrating computer systems — where most MSPs, systems integrators and VARs-with-services self-classify; 541511 = writing custom code to spec (dev shops); 541513 = ongoing management of a client's OWN facilities/data-center operations (outsourced IT ops); 541519 = residual services (software installation, disaster recovery). MSP benchmarks (Kaseya, Service Leadership) and deal trackers sample by business model, not NAICS — expect their populations to span 541511/541513/541519 and treat NAICS-coded government data vs. industry surveys as different frames.
- Revenue-mix normalization trap: small-MSP receipts blend high-margin MRR, low-margin hardware/license resale (pass-through inflates revenue, deflates labor_share — the record's 0.6444 reflects this) and lumpy project work. Multiples are quoted on adjusted EBITDA of MRR-weighted books; a resale-heavy firm at the same revenue is worth half as much. Never apply MRR-book multiples to blended-revenue firms.
- EBITDA normalization: owner-operator comp inside SDE for sub-$2M-EBITDA MSPs; Service Leadership's own data shows best-in-class ~2.5x median EBITDA% — wide dispersion means 'average margin' hides the target-selection reality.
- Per-seat MRR is a capture-POSITIVE pricing structure near-term (fees decoupled from hours), unusual in sector 54 — but model the seat-count channel explicitly in pi_dist: AI-driven client headcount reduction shrinks the billing base even if rates hold.
- The 2022 receipts vintage predates both the AI service-desk products and the 2024-2025 deal-tempo peak; check current (2025-2026) deal pace and pricing rather than extrapolating the dataset era.

## Frozen formulas — Python calculates them

- `V_raw = labor_share × ai_replaceable_share`
- `V = 10 × min(1, V_raw / 0.25)`
- `C = 10 × pi_dist × pi_moat`
- `A = 10` when `t50_years=0`, otherwise `min(10, 10 / t50_years)`
- `TD = clamp(log10(n_band) / 4, 0, 1)`
- `OW = clamp((owners_60plus_pct − 0.10) / 0.35, 0.1, 0.9)`; add 0.1 for documented succession shortage, cap at 1
- `CFD = min(0.9, log10(1 + active_consolidators) / 2)`
- `B = 10 × sqrt(TD × OW) / (1 + 0.3 × CFD)`
- `M = clamp(4 × (exit_mult − buy_mult) / buy_mult, 0, 10)`
- `S = (V × C × A × B × M)^(1/5)`
